- **目的**: 新しいステージJSONを作る際、このドキュメントのA章だけ読めば事故らず作れる状態にする
- **構成**: A(確定事実) / B(未確認・要注意) / C(未実装の設計方針・ロードマップ) を厳密に分離している

## ★ ステージ設計方針（データ駆動・交戦地点主導）

**確立: 2026-07-04 ／ このドキュメント最重要章**

> 設計の根本思想をまとめる。各セクション（A〜H）の詳細はあくまでこの方針を実現するためのツール。

---

### ★-1. 基本思想

- **交戦地点を先に決め、spawnはそこから逆算して配置する**
- スポーン地点・時刻・敵種は JSON（spawnEvents）で完全制御する。エンジンに任せる要素を減らす
- 難易度は「HP」よりも「いつ・どこから・どの種類が来るか」で調整するほうが直感的で事故が少ない
- 理不尽（E-11/E-14参照）を避けることが、難しさより優先度が高い

---

### ★-2. 交戦地点主導（設計の順序）

```
① パスを引く（護衛の移動ルートを決める）
② 「ここで戦わせたい」交戦地点を3〜5点決める
③ その地点に届くタイミングでspawnするには t=? 秒か計算する
④ spawnをその地点から十分離れた位置に置く（★-3参照）
⑤ spawnEvents に書く
⑥ シミュレーター or 実機で検証（★-5参照）
```

wave方式と違い、「いつ出てくるか」が秒単位で直接指定できるため、手順③④のPDCAが格段に速い。

---

### ★-3. spawnは遠方に置く

- **spawn地点は交戦地点から離れるほど安全**。近すぎるとE-11（スポーン即衝突）が起きる
- ルール: **spawn → 交戦地点のFlowField距離が、護衛speed × 2秒分以上あること**（例: speed=40 → 80px = 0.8セル以上）
- 前方（護衛進行方向の先）にspawnを置く場合は E-15参照。少しずらす原則を適用すること
- `spawns` dict に名前をつけておくと、複数イベントで同じ地点を再利用できて見通しが良い

---

### ★-4. 難易度調整の優先順位

同じ「難しくする」でも、調整手段によってゲーム感覚への影響が異なる。

| 優先度 | 手段 | 効果 | 注意 |
|---|---|---|---|
| ① | spawnEventsの`time`を早める | 護衛と早く交戦 → プレイヤーの準備時間が減る | E-11に注意 |
| ② | `count`を増やす・`interval`を縮める | 連続ラッシュ | タワー飽和に注意 |
| ③ | `speed`を上げる | 逃げ足が速く倒しにくい | E-16（シム差）に注意 |
| ④ | `hp`を上げる | 単純に硬くなる | E-16に注意。効果が鈍い（★-4末尾参照） |
| ⑤ | 敵種を重い種（worker/police）へ変更 | 質的難易度変化 | 新規交戦地点設計が必要なことも |
| ⑥ | buildSpotを減らす | プレイヤー側の手数が減る | バランス破壊リスク大 |

> HPを上げても難しさが上がりにくい理由: タワーのDPS合計が一定なら、HPが増えた分だけ交戦地点が前方にずれるだけで、本質的な難度変化が少ない（E-16も参照）。

## ★A. フォーマットA: 直線交戦テンプレート（2026-07-05 確立）

> ★章の設計思想を「実際に計算が閉じる」形まで落とした最初の具体フォーマット。全ステージに使う型ではなく、再利用可能なカードの1枚。

### ★A-1. 位置づけ

交戦地主導モデルを実計算まで落とした最初の設計フォーマット。マップ形状は問わず、下記5条件を満たす配置なら成立する。

### ★A-2. 成立条件

1. **交戦地は path の曲がり角に置く** — 護衛が必ず通る確定点であり、折れることで護衛の動きが読みやすい
2. **spawn は交戦地に対して「path直線上」か「path直交線上」に置く** — leashTo への経路が直線になり、障害物回避の迂回が入らず、`距離 ÷ 敵speed` で到達時刻が一撃で出る
3. **待機付き leashTo で交戦地を座標確定する** — 敵は leashTo（= 交戦地）到達後、護衛が同セルに来るまで待機。spawn をいくら遠ざけても交戦地が1ミリも動かない
4. **2種類の猶予が独立に回せる**
   - `spawn距離` = 接近猶予（敵が交戦地へ向かう道中の迎撃時間）
   - `spawn_time` = 待機猶予（敵が交戦地で護衛を待ち構える時間）
   - 両方とも護衛 speed に非依存
5. **spawn→leashTo の直進経路が護衛 path と2セル以内で並走・交差しないこと** — 交差すると敵は視界内の護衛を無視して交戦地へ直進する（leashTo 仕様通りの挙動だが、プレイヤー体験として破綻）。マップ設計時点で回避すること。

   **チェック方法**: spawn 座標と leashTo 座標を結ぶ直線上のセルを列挙し、そのセルまたは隣接セル（距離≤2）が護衛 path に含まれていないことを確認する。

### ★A-3. このフォーマットが解決するもの

| 問題 | 解決策 |
|---|---|
| 交戦地が速度差やFlowFieldズレで動く | leashTo で座標固定 → 護衛speed変更後も全ステージ作り直し不要 |
| spawnが交戦地に近すぎて猶予ゼロ | 直線＋待機で遠くに置いても確定。距離をそのまま猶予に使える |
| 追走シミュレーションの不確実性 | 直線経路なので `距離÷speed` で計算が閉じる |

### ★A-4. 適用範囲外

- spawn→交戦地の間に prop や空き地の迂回が挟まり、経路が直線にならない配置 → 追走の不確実性が戻る
- 交戦地を角以外（直線の途中）に置きたい場合 → 成立はするが護衛の動きの読みやすさが落ちる

### ★A-5. 前提となる未実装項目

**待機付き leashTo**（spawnEvent に `leashTo:{col,row}` を追加、敵は到達後護衛が来るまで待機）。CCが実装可能と回答済み（ghost 2秒維持・逆算に+2秒・イベント単位・直線経路前提）。

time 逆算式：
```
T_spawn = T_encounter - T_travel - 2(ghost) - 待機バッファ(任意)
T_travel = 直線距離(セル数) × 100px ÷ 敵speed(px/s)
```

待機付き leashTo 実装後は「護衛到達より少し前に着く」程度で十分。精密逆算は不要になる。

## A. エンジン確定事実（CC確認済み）

### A-1. 座標・サイズ

- **CELL = 100px**（`config.js:2`）※旧メモに「64px」とあるのは廃止済み仕様。今後は100px基準で統一
- **grid（cols/rows）は完全自由**。`GameScene.js:16-19`でJSONの値を直接`COLS/ROWS`に代入しているだけで、推奨値（例: 25×8）は目安に過ぎず制約ではない
- 護衛の移動時間の計算式：
  ```
  距離(px) = (pathの区間数) × 100
  到達時間(ms) = 距離(px) ÷ speed(px/秒) × 1000
  ```

### A-2. escorts / path

- `path`は隣接セルを1マスずつ移動するウェイポイント配列。**対角移動不可**
- `goal`は必ず`path`の最後の要素と同じ座標にすること（一致しないと護衛がゴールしない）
- `start`は`path`の最初の要素と一致させる

### A-3. startTrigger（複数護衛のリレー）

- **`startTrigger: "afterPrevious"`は現状未実装。JSONに書いても無視される**
- 実際の挙動：**前の護衛がゴール到達 or 死亡した瞬間から固定4秒（`RELAY_INTERVAL = 4000ms`）後に次の護衛が自動スタート**。タップでスキップ可能
- ⚠️ 「前の護衛の生死によって次のリリース条件が変わる」といった設計は**現状不可能**。複数護衛ステージの難易度設計はこの制約を前提にすること

### A-4. waves（暫定仕様・移行中）

> ⚠️ **【2026-07-05 重要方針転換】wave方式は spawnEvents 方式へ置き換え済み（stage_01.json 移行完了）。**
> 本A-4のwave時間計算式（duration/startDelay/spawnInterval）は、off-by-one・最終wave無限湧き等の事故を構造的に生む欠陥があり（E-8/E-9参照）、廃止する方向。
> **新規ステージ設計は「★ ステージ設計方針」章および A-4-new（spawnEvents）を前提とすること。**
> 本節は旧ステージ（stage_02.json 等）の読解・移行作業のための参照用として残す。

各waveのフィールド（旧方式・参照用）：

| フィールド | 説明 |
|---|---|
| `startDelay` | wave開始からの待機時間(ms) |
| `duration` | このwaveの継続時間（**秒**単位、必須。省略厳禁） |
| `spawnInterval` | グループ間の間隔(ms) |
| `groupSize` | 1グループの人数。**省略時デフォルト1**（`wave.groupSize ?? 1`） |
| `groupInterval` | グループ内の個体間隔(ms) |
| `enemy` | `{type, hp, speed, damage, reward}` |

**やってはいけないこと**：
- `count`フィールドは使わない → 実際のスポーン数を制御しない。durationの自動計算にのみ影響し、`groupSize`未指定だと1体ずつのグループになってしまう
- `duration`の省略 → 後半グループが出ない、または（wave内で）護衛到達まで無制限スポーンし続ける不安定な状態になる

**時間軸の計算式（★最重要・今日何度も訂正が入った箇所）**

1. **`spawnInterval`の起点**：「あるグループの全個体スポーン完了 → 次グループ開始」の間隔（グループ開始〜開始ではない）。`WaveManager.js:85-86`で確認済み

2. **`duration`と`startDelay`は直列**（同じ基準点からの並行カウントではない）：
   ```
   wave開始（前waveの終了 or ステージ開始）
     → startDelay 待機
     → elapsed = 0 スタート
     → duration 秒後、このwaveは終了
   ```
   ⚠️ **重要**: wave[1]以降のstartDelayは「そのwave内の最初のグループ出現を遅らせる」だけであり、
   waveの終了タイミング（elapsed基準のdurationカウント）には影響しない。
   startDelay が duration に近い値だと、wave内でほとんどゾンビが出ない可能性がある。

3. **wave内、numGroups（グループ数）を指定するフィールドは存在しない**。グループはelapsed=0基準で `spawnInterval` 間隔に発火し、**「発火判定時のelapsed < duration×1000」なら発火する**。つまりnumGroupsは`duration`と`spawnInterval`から決まる。設計者が「何グループ出したいか（N）」を決めたら、以下の**上下両側の不等式**で`duration`を挟む：
   ```
   groupSize=1 の場合（発火はelapsed = 0, si, 2si, ... で起きる）:
     N番目のグループ発火時刻 = (N-1)×spawnInterval
     N+1番目のグループ発火時刻 = N×spawnInterval

     (N-1)×spawnInterval < duration×1000 ≦ N×spawnInterval
   ```
   ⚠️ **【最重要・2026-07-02にoff-by-oneで事故】** 上限側（`≦ N×spawnInterval`）を必ず守ること。
   duration がこれを超えると **N+1番目のグループが「duration境界の直前」で駆け込み発火し、意図より1体多く湧く**。
   例：spawnInterval=5000, N=4体狙い → `15000 < duration×1000 ≦ 20000`、つまり **duration=16〜20秒**。
   誤ってduration=24秒にすると、elapsed=20000ms<24000msで5体目が湧いてしまう（実際にこれで+1体事故）。
   groupSize≧2の場合は各グループ内の最終個体時刻 `(groupSize-1)×groupInterval` も加味すること。

4. **wave全体の絶対終了時刻（wave indexが切り替わる時刻）**：
   ```
   ゲーム開始を t=0 として、

   wave[0] が終了する絶対時刻 = startDelay₀ + duration₀×1000
   wave[1] が終了する絶対時刻 = startDelay₀ + (duration₀ + duration₁)×1000
   wave[n] が終了する絶対時刻 = startDelay₀ + Σduration[0..n]×1000
   ```
   ※startDelay は wave[0] のものだけが累積に含まれる。wave[1]以降の startDelay は
   あくまで「そのwave内で最初のグループが出るまでの待機」であり、wave終了タイミングには加算されない。

   ⚠️ **【最重要・最終waveの罠】** `_calcWaveIdx()`のループが `waves.length - 1` までしか回らないため、
   **最終wave（配列の最後）はdurationで終了しない。護衛が到達（escort.reached）するまで無限に発火し続ける。**
   最終waveの`duration`値は実質無意味。よって最終waveの敵数は「発火間隔」と「護衛到達までの残り時間」から自動的に決まる。
   設計時は「最終waveは護衛到達まで敵が途切れず湧く」前提で組むこと（敵数を厳密に固定したいなら最終waveの後にダミーの空waveを置く等の工夫が要るが、通常は割り切ってよい）。
   最終waveの実発火数 = floor((護衛到達時刻 − 最終wave開始時刻) ÷ spawnInterval) + 1

5. **護衛到達前に全waveが湧き終わるかの検算は必須**：
   ```
   startDelay₀ + Σduration[全wave]×1000 < 護衛到達時間(ms)
   ```
   これを満たさないと、後半の敵（特に最終waveの終盤）が出現しないまま切り捨てられる

6. **現状の難易度設計方針（Plan A）**：wave内の敵は全員同じ強さ。escalation（強くなる）はwave間の敵種切り替えでのみ行う。wave内で敵を強くする設計（Plan B）は現状シミュレーターなしでは危険（C-3参照）

7. **【実装済 2026-07-04】spawn直書きによる stat 上書き**：

   `WaveManager.js` のspread順を修正し、優先順位が確定：

   **ZOMBIE_BASE（最低） < wave.enemy < spawn の直書き値（最高）**

   3パターンが使える：
   ```json
   // パターンA: wave.enemy に type + 数値（waveごとに種と強さを制御）
   "enemy": { "type": "police", "hp": 50, "speed": 50, "damage": 15, "reward": 20 }

   // パターンB: spawn.type でスキン固定 + wave.enemy で強さ制御
   "zombieSpawns": [{ "col": 0, "row": 1, "type": "police" }]
   "enemy": { "hp": 50, "speed": 50 }

   // パターンC: spawn 直書きで完全上書き（最高優先）
   "zombieSpawns": [{ "col": 0, "row": 1, "type": "police", "hp": 50, "speed": 50 }]
   ```

   **これにより「見た目（スキン）は種類豊富・強さは設計者が自由設定」が両立可能になった。**

   - wave 内で敵種は混ぜられない：1つの wave は `enemy` を1オブジェクトしか持てない。「雑魚の中に時々硬いの」は wave を交互に切り替えることで実現する（wave交互推奨・安全）。

### A-5. zombieSpawns

- 護衛が近づくたびに**「護衛から最も近いspawn地点」がFlowField距離で自動選択**される（ランダムでも順番でもない）
- 1グループの敵は全員同じ1箇所から連続して湧く
- **制約**：ゴール地点・path上のセルに置かない
- **スポーン後2秒の幽霊状態**（`Zombie.js` `_spawnTimer=2000`）：スポーン直後の2000ms間、ゾンビは完全無敵（移動不可・護衛への攻撃不可・タワーからの被弾不可、alpha0.4の半透明表示のみ）。2秒経過後に通常のゾンビとして動き出す。プレイヤーへの視認猶予として機能する良い仕様だが、難易度計算では「敵の実効的な脅威開始が湧きの2秒後」になる点に注意（G-5参照）

### A-7. props（当たり判定あり・footprint注意）

- **propsは自動的に`obstacles`と同等の当たり判定を持つ**（`GameScene.js:54-61`）。`obstacles`配列への二重登録は不要
- タワー設置時も同様にブロックされる（`GameScene.js:946`）
- **footprint**：JSON上は左上1座標のみ指定。実際のブロック範囲は`PROP_DEFS`の`cols × rows`分、全セルが自動的に塞がれる

  | type | cols×rows |
  |---|---|
  | shop | 4×4 |
  | shop_front / shop_back / house_front / house_back / 家青 / 家緑 / アイスクリーム / ハンバーガー | 3×3 |
  | guardrail_h / guardrail_v / tree / 木1 / 木2 / ベンチ | 1×1 |
  | 公園 | 6×6 |
  | 駐車場 / 駐車場車有り | 4×2 |
  | スーパーマーケット | 6×3 |
  | 商店街 / house5_front / house5_back | 15×3 |

  ※新しいtypeが追加された場合は都度CCに`PROP_DEFS`の値を確認すること

- **⚠️既知の事故パターン**：**props配置に対するバリデーション（衝突警告）は一切ない**。path上やbuildSpots上にpropsのfootprintが被っても無警告でそのまま動く → 護衛が経路上でブロックされてフリーズ、buildSpotsが使用不可になる等のバグが発生する。**propsを置く際は必ずpath全座標・buildSpots全座標とのfootprint衝突を手動（または計算で）確認すること**

### A-8. ground_cells

- 道路などのテクスチャ指定のみ。**当たり判定には一切影響しない**（見た目だけ）

### A-9. minSurvivors / クリア・敗北判定

- 全護衛の移動が終了した時点で、`survivors（ゴール到達数） >= minSurvivors` ならクリア
- **即敗北判定あり**：道中で「残っている護衛が今後全員ゴールしても`minSurvivors`に届かない」と判明した瞬間、即座に敗北になる（ゲームが最後まで進行しない）

### A-11. 当たり判定とタワー設置の確定ルール（2026-07-04 CC確認済み・最重要）

今日まで曖昧だった「何が通行不可で、どこにタワーを置けるか」がコード確認で完全に確定した。
**この2つは独立したルールであり、混同すると設計が根本から破綻する。**

#### 通行可否（ゾンビ・護衛が通れるか）
`FlowField.js` の `isWalkable()` が false を返すのは **2つだけ**：
1. グリッド外
2. `pf.blocked` に登録されたセル（= `obstacles[]` + **props の footprint** + 外周ガード）

- **`ground_base` / `ground_cells` のタイル種別（grass / 道路横 / road_vertical / asphalt）は当たり判定に一切影響しない。全部通行可。**
- ⚠️ **つまり「grass は壁になる」は誤り。** ゾンビも護衛も grass セルを自由に通れる。
- **通行不可にできるのは props（建物・木など footprint を持つもの）だけ。**
- 【今日の事故】「道以外を全部 prop で埋めて道を1本に絞る」設計をした際、grass が壁だと勘違いして「空き地（grass）を壁として使う」案を出しかけた。grass は通行可なので壁にならない。**道を絞りたいなら道以外を prop で物理的に埋めるしかない。**

#### タワー設置可否（buildSpot にタワーが建つか）
`_canPlace()` の条件（全て満たす必要あり）：
1. グリッド内
2. `isWalkable()` = **prop も obstacle も外周ガードも踏んでいない**
3. `buildSpots` に登録されている（空配列なら全マスOK）
4. 同セルにタワーが無い
5. 所持金が足りる

- **`ground_cells` のタイル種別は設置条件に含まれない。** grass 上でも道路タイル上でも、**prop さえ被っていなければタワーを置ける。**
- ⚠️ **buildSpot を prop の footprint 上に置くとタワーは設置できない。**（条件2で弾かれる）
- 【今日の事故】buildSpot を建物 footprint 内に置いた → 設置不可で全滅。**buildSpot は必ず prop footprint 外に置くこと（A-6 の制約は正しかった）。**

#### この2ルールの帰結（道幅の設計）
- 道幅1マスで「道以外を全部 prop で埋める」と、**タワーを置く空きセルが道路（=path）しか残らず、path 上には buildSpot を置けない**ため詰む。
- 解決策：**道を2マス幅にする**（片方 path・もう片方を buildSpot 用の通行可セルにする）。両方 grass/道路で通行可のままでよい。壁は上下を prop 帯で挟んで作る。
- ⚠️ ただし道2マス幅は「ゾンビもその2マス幅を自由に動ける」ことを意味する（両方通行可のため）。1本道の追走を厳密にしたい場合の副作用に注意。

### D-4. プレイログ機能【実装済 2026-07-02】

  |---|---|
  | `[SPAWN]` | t / wave番号 / spawn座標 / hp / 累計スポーン数(total) |
  | `[HIT]` | t / 護衛被弾 / damage / 護衛HP(前→後) |
  | `[KILL]` | t / wave番号 / 累計撃破数(killCount) |

- **出力例**：
  ```
  [SPAWN]  t=8000ms  wave=0  spawn=(0,0)  hp=110  total=1
  [HIT]    t=35000ms  護衛被弾  damage=18  護衛HP: 100→82
  [KILL]   t=41000ms  wave=0  killCount=3
  [REACH]  t=110000ms  護衛ゴール到達  生存護衛=1
  ```

- **Claudeがログから読み取れること**：
  - `[SPAWN] total`の最終値 = 実スポーン数 → 設計値とのズレ（off-by-one、最終wave無限湧き）が即判明（A-4参照）

  - `[RESULT] すり抜け` → タワーの処理能力を超えた敵の総量

## E. 既知の落とし穴（今日発生した実例ログ）

1. **セルサイズの旧仕様混在**：メモリ上「CELL=64px」が残っていたが実際は100px。数値計算前に必ずconfig.jsの現在値を疑うこと
2. **`count`フィールドの罠**：wave定義に`count`があっても実スポーン数を制御しない。`groupSize`なしだと1体ずつの弱いグループになる、またはduration次第で無制限スポーンになる
3. **`spawnInterval`の意味誤読**：グループ開始〜開始の間隔だと誤解しやすいが、正しくは「前グループの湧き終わり→次グループ開始」
4. **`duration`と`startDelay`の関係誤読**：同じ基準点から並行カウントだと誤解しやすいが、正しくは直列（startDelay消化後にelapsed=0）
5. **grid10×8で街が窮屈になった事例**：チュートリアルステージでも街としての説得力を求めるなら15×10クラスが実質的なミニマムサイズだった（stage_01の反省）
6. **propsの無警告衝突リスク**：footprintの概念を知らずに配置すると、経路ブロック等の重大バグが無警告で発生する
7. **path区間数の数え間違い**：長いpath（40区間超）を目視でフェーズごとに数えると1区間分ズレやすい（今日、43区間と数えたが実際は44区間だった）。護衛到達時間がズレるとwave設計全体の検算が狂うため、区間数は`path.length - 1`で機械的に数えること（目視カウントに頼らない）
8. **durationのoff-by-oneで+1体**：「duration ≧ 最終スポーン時刻」だけ守っても、上限を超えると次グループが駆け込み発火する。設計値4体・spawnInterval5000・duration24秒 → elapsed=20000ms<24000msで5体目が湧いた実例あり。**必ず `(N-1)×si < duration×1000 ≦ N×si` の両側で挟む**（A-4-3参照）
9. **最終waveが護衛到達まで無限湧き**：`_calcWaveIdx()`が`length-1`までしか判定しないため、最終waveはdurationを無視して護衛到達まで湧き続ける。設計値12体のwave2が実際は14体湧いた実例あり。最終waveの敵数は「発火間隔と護衛到達までの残り時間」で決まると理解して設計すること（A-4-4参照）
11. **【E-11】Z字（横ミアンダ）1本道では「追走（追いかけっこ）」は原理的に作れない**：spawn をどこに置いても、**護衛が往復するЗ字では、護衛がその spawn 地点に近づくフェーズが必ず来る** → その瞬間 spawn から湧いた敵と「正面衝突」になり、回避不能被弾で即死する。純粋な「背後からの追走」を保証できる spawn はスタート地点1箇所のみ。【実プレイ実証】spawn を4箇所（折り返しの角含む）に置いた15×15ステージで、護衛が折り返しの角(14,1)に近づいた瞬間、そこから湧いた敵の正面に突っ込み **t=29〜34秒で HP110→0 の即死**。2回とも同じ場所で全滅。**教訓**：「追ってくる」体験が欲しいなら、Z字1本道ではなく **分岐のある導線**（敵が別ルートから回り込める）が必要。1本道では spawn を背後1箇所に限定するしかなく、「後方から押し寄せる密度」で緊張を作ることになる。

12. **【E-12】均一な敵だけでは難易度が「断崖」になり「なんとかクリア」の帯が作れない**：敵が全員同じ強さだと、難易度が「タワーN基で全部倒せる / 倒せない」の二値になり、**HP を削られつつ生還する中間帯（★1〜★2の薄氷クリア）が存在しない。**【シミュレーション実証】spawn 背後1箇所・均一敵で密度を振ったが、結果は「良配置=ノーダメ HP100 / 基数不足=即死 HP0」の断崖のみ。**教訓**：「あーなんとか迎撃！」は敵の平均強度ではなく **強弱の分散** で作る。雑魚を捌くリズムの中に、時々タワー火力を超える硬い敵を混ぜ、「抜けそう→ギリ間に合った」を作る。→ A-4-7 の「wave 交互で種を変える」で実装する。

13. **【E-13】シミュレーション無しで数値を出すのは「当てずっぽうに体裁を付けただけ」**：衝突チェック・到達時間の検算・敵数計算が全部通っていても、**「そのステージが実際にクリア可能か」「難易度が適切か」は別問題**。今日、検算 OK のステージが実プレイで即死した。**教訓**：コード実行環境があるなら、ステージを出す前に **必ずシミュレーターで「クリア可能か」「難易度帯に収まるか」を回す**（H章）。

14. **HP増だけでは難易度が上がりにくい**：敵speedが半減している環境ではタワー射程内の滞在時間が2倍になり、敵HPを1.5〜1.8倍にしてもタワーが処理しきってノーダメになる（v3で実証：helmet HP60→110でも4基ノーダメ）。1体ずつ湧く限り処理が溢れないため、難易度を上げるにはHPよりも`groupSize`（同時出現数）を増やして瞬間的な処理能力オーバーを作る方が効く

15. **【E-14】「難しい」と「理不尽」は別物**：ゲームとして許容される難しさは「攻略の余地がある難しさ」のみ。「理不尽」はプレイヤーに攻略の余地を与えない難しさで、繰り返しプレイ意欲を破壊する。見分け方: 同じ負け方を2回以上した場合、それは理不尽の可能性が高い（1回は「不運」だが2回は「構造的問題」）。【実例】stage_01 の E-11: police が (7,13) から湧いた瞬間、護衛が既に row 13 を歩いており正面衝突 → `t=114.9s` で被弾。このspawn位置は護衛が row 13 に到達する前は安全だが、到達後は毎回正面衝突になる。プレイヤーにはどうにもできない。**教訓**: 「難易度調整」と「理不尽除去」は独立した作業として行うこと。まず理不尽を潰してから、難易度を上げる。

16. **【E-15】前方spawnの「少しずらす」原則**：護衛の進行方向の前方（護衛が向かっている先）にspawnを置くと、湧いた瞬間に護衛と向かい合わせになり E-11 になる。対策は「spawn を進行方向の軸からずらす」こと。**ずらす方向**: ①進行方向と直交する方向（横）にずらす、②進行方向の「後ろ側」にずらす、の2択。【検証実例】stage_01 の spawn B=(7,13): row 13 上に spawn があるため、護衛が row 13 フェーズ（t=72s〜）に入ると正面衝突 E-11 が発生。(5,12) や (9,14) のように行をずらすと FlowField が col 7 付近でも安全な経路を通り、交戦地点を row 13 の側道まで引き寄せられる（シミュレーション確認済み）。**ルール**: 前方 spawn を置く場合は、**護衛の進行方向の軸から最低1セル以上ずらすこと**。軸上に置いてよいのは「護衛が絶対にそこを通らない方向」のみ。

17. **【E-16】シミュレーターは実機よりタワーが強い**：シミュレーターのタワー攻撃処理は実機(Phaser/GameScene)と完全一致しておらず、タワーが過剰に強く出る傾向がある。【実証: 2回確認済み】①sim2.js で hp=80 CLEAR 判定 → 実機でも CLEAR だが 1 発被弾（hp=40 残存）。②シミュレーターで GAMEOVER になったケースが実機でギリギリ CLEAR になるケースが複数あった。**影響**: シミュレーターが「GAMEOVER」と言っても実機では生き残る可能性がある。逆に「CLEAR」でも実機で数発食らうことはある。**使い方の指針**: シミュレーターは「絶対無理な配置を切る」フィルターとして使うのが正しい用途。「シムでCLEAR＝実機でCLEAR」と信頼しすぎない。最終検証は必ず実機で行うこと。

### G-5. 難易度レバーの効き序列（2026-07-02 実測で判明）

speed半減環境（護衛40・敵27.5〜35）では、**敵の射程内滞在時間が長くなり、多くの難易度調整が「滞在時間の余裕」に吸収される**。3回の実測（v2速度/v3のHP増/v4のgroupSize増）で全てノーダメだったことから、以下の効き序列が判明：

| レバー | 効き | 理由 |
|---|---|---|
| タワー`fireRate`延長 | **強** | 滞在時間内の手数を直接削る。滞在時間の有利を打ち消す唯一の直接手段 |
| タワー`range`短縮 | **強** | 滞在時間そのものを削る |
| タワー購入価格↑ | **強（副作用小）** | 「いつでも強タワー」を防ぐ。強さを殺さず依存度だけ下げる |
| 敵`speed`↑ | 中 | 相対速度差を詰めるが「がちゃがちゃ感」再発リスク |
| startMoney↓ | 中 | ただし本作はタワー売買が基本運用（下記）なので、絞りすぎると機動的運用の楽しさを削ぐ |
| 敵HP / damage / groupSize | **弱** | 滞在時間に吸収されやすい。単体では効きにくい（複数レバー併用時のみ有効） |

- ⚠️ **スポーン後2秒の幽霊状態（A-5補足）**：ゾンビはスポーン後2000ms完全無敵（移動・攻撃・被弾すべて不可、半透明表示）。この2秒＋低速移動が「処理猶予」を生み、groupSize増が効かない一因。プレイヤーの視認猶予として良い仕様だが、難易度計算では敵の実効的な脅威開始が2秒後ろ倒しになる点に注意
- **タワー売買が基本運用**：本作はタワーを設置しっぱなしにせず、護衛の移動に合わせて売却→再設置する動的運用が前提（`sell`列＝売却額）。エスコート型TD特有の「守るべき場所が時間で動く」性質に対応。所持金の潤沢さ＝置き直しの自由度なので、難易度レバーとして所持金を使う際はこの楽しさとのトレードオフに注意

### G-6. タワー性能調整方針（2026-07-02 議論、未実装）

- 目指すのは「狙撃を弱くする」ではなく「**他タワーに狙撃にない固有役割を与える**」方向（KRのタワー役割分化を参考）。狙撃が複数レーンをカバーできる強さ自体はTDの核心的な面白さであり、否定しない
- KR流の王道は「**長射程＝低速・単体特化 / 短射程＝高速・数処理**」。狙撃無双を崩すなら、damage削減より`fireRate`延長で「手数を奪う」方がジャンル的に正しい
- ただしfireRateを延ばしすぎると「撃たない待ち時間」がストレスになるため、極端にはしない
- 具体案として挙がった狙撃調整（いずれか単独から試す。同時適用は過剰弱体化リスク）：
  1. range 4.5→3.0〜3.5に短縮（2.0まで一気に詰めると狙撃自体が死にタワー化するので段階的に）
  2. 購入価格 200→350〜400（最優先・副作用最小。「いつでも狙撃」を防ぐ）
  3. damage弱体化で3〜4発撃破に（単体では弱いが1と併用で効く）
- **推奨手順**：まず案2（価格）だけ適用してプレイ→まだ狙撃一強なら案1か3を一つ足す（変数を同時に動かさない）

### G-7. タワー2種限定フェーズ（2026-07-02 決定）

- **難易度計測フェーズの間は、タワーを「通常(basic)」「狙撃(sniper)」の2種のみに限定する**
- 理由：タワー7種あると「どのタワーを何基置いたか」でプレイ結果がブレて、ステージ側の難易度なのかタワー選択なのか切り分けられない。変数を減らして難易度を確定させるため
- basic（range2.5/fireRate900/dmg25＝手数・前線処理）と sniper（range4.5/fireRate1600/dmg調整中＝一撃・複数レーンカバー）の最小対比構成でステージ難易度を測る
- 残り5種（rapid/laser/punch/bat/cannon）は、この2種の基準が固まった後に「2種に対してどういう固有役割か」を持たせて1つずつ追加する

### G-8. 基準値まとめ（2026-07-04 確定）

- **護衛 speed: 40**（基準・変更なし）
- **敵 speed: 50**（護衛40に対し1.25倍。「追われる」速度差を作るため。旧「各種半分」の方針から変更）
- **タワー実値（balance.json 確認済み）**：

  | | cost | range | fireRate | damage |
  |---|---|---|---|---|
  | basic | 100 | 2.5セル(250px) | 900ms | 25 |
  | sniper | 200 | 4.5セル(450px) | 1600ms | 40（config.jsの70より低い実値） |

- **弾**：speed 500px/秒・ターゲット追尾ホーミング・距離10px未満で着弾（レーザーのみ即着弾ビーム）
- **移動方向**：FlowField BFS 4方向のみ（対角移動なし）
- ⚠️ G-7 のタワー2種限定フェーズは継続中。basic/sniper のみで難易度を測る。

## H. ステージ難易度シミュレーター（2026-07-04 構築）

### H-1. 存在と目的

ステージJSONを出す前に「クリア可能か」「難易度が狙った帯（良配置で薄氷クリア／雑配置で敗北）に収まるか」を**動的に検証する**ためのツール。机上の検算（到達時間・敵数）では絶対に分からない「実際に勝てるか・面白い難易度か」を、出荷前に数値で確認するためのもの。

### H-2. 再現しているエンジン挙動

- 護衛の移動（path・speed・時刻→座標の線形補間）
- 敵の spawn スケジュール（wave の startDelay/duration/spawnInterval/groupSize/groupInterval、最終wave無限湧き）
- 敵の追跡（FlowField・BFS 4方向・対角なし・prop 回避）
- スポーン後2秒の幽霊状態（無敵・不動）
- タワーの射撃（range/fireRate/damage、射程内で護衛に最も近い敵を狙う）
- **経済（startMoney・撃破 reward・買える順にタワーを順次設置）** ← これが無いと「序盤の金欠で崩れる」を見逃す
- 護衛 HP・被弾・クリア/敗北判定

### H-3. 使い方

1. ステージの path / blocked(prop footprint) / spawns / waves / towers を設定
2. **複数のタワー配置パターン**（良配置・中配置・雑配置）で回す
3. 判定基準：
   - **良配置 → HP 20〜50% 生還**（★1〜★2の薄氷）が理想。ノーダメ生還＝簡単すぎ、即死＝理不尽
   - **雑配置 → 敗北**（考えないと勝てない＝緊張がある）
   - この差が「プレイヤーの工夫の余地」＝面白さ
4. spawn位置・敵種構成・敵数・タワー価格を振って、上記の帯に収まる設定を探索してから JSON 化

### H-4. シミュレーターの限界（過信しない）

- 敵の護衛攻撃モデルは簡略（隣接で1秒ごとに damage）。実エンジンの攻撃頻度・範囲と差がある可能性。
- タワー配置は人間が決め打ちしており、実プレイヤーの動的な売買運用は再現していない。
- **必ず「シミュレーション予測 → 実プレイ1回 → ログと照合」でキャリブレーションすること。** ズレていればシミュレーターを補正する。合っていればシミュレーターを信頼して量産に使える。

### H-5. ワークフローへの組み込み

D-1 のステップ5（wave設計）の後に「**5.5. シミュレーターでクリア可能性と難易度帯を検証（H章）。狙った帯に入るまで数値を振る**」を追加する。実プレイ（ステップ6）は「体感の確認＋シミュレーターのキャリブレーション」の位置づけになる。
