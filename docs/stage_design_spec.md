## A. エンジン確定事実（CC確認済み）

### A-1. 座標・サイズ

- **CELL = 100px**（`config.js:2`）※旧メモに「64px」とあるのは廃止済み仕様。今後は100px基準で統一
- **grid（cols/rows）は完全自由**。`GameScene.js:16-19`でJSONの値を直接`COLS/ROWS`に代入しているだけで、推奨値（例: 25×8）は目安に過ぎず制約ではない
- 護衛の移動時間の計算式：
  ```
  距離(px) = (pathの区間数) × 100
  到達時間(ms) = 距離(px) ÷ speed(px/秒) × 1000
  ```

### A-2. escorts / path

- `path`は隣接セルを1マスずつ移動するウェイポイント配列。**対角移動不可**
- `goal`は必ず`path`の最後の要素と同じ座標にすること（一致しないと護衛がゴールしない）
- `start`は`path`の最初の要素と一致させる

### A-3. startTrigger（複数護衛のリレー）

- **`startTrigger: "afterPrevious"`は現状未実装。JSONに書いても無視される**
- 実際の挙動：**前の護衛がゴール到達 or 死亡した瞬間から固定4秒（`RELAY_INTERVAL = 4000ms`）後に次の護衛が自動スタート**。タップでスキップ可能
- ⚠️ 「前の護衛の生死によって次のリリース条件が変わる」といった設計は**現状不可能**。複数護衛ステージの難易度設計はこの制約を前提にすること

### A-4. waves（最重要・計算ミスが起きやすい箇所）

各waveのフィールド：

| フィールド | 説明 |
|---|---|
| `startDelay` | wave開始からの待機時間(ms) |
| `duration` | このwaveの継続時間（**秒**単位、必須。省略厳禁） |
| `spawnInterval` | グループ間の間隔(ms) |
| `groupSize` | 1グループの人数。**省略時デフォルト1**（`wave.groupSize ?? 1`） |
| `groupInterval` | グループ内の個体間隔(ms) |
| `enemy` | `{type, hp, speed, damage, reward}` |

**やってはいけないこと**：
- `count`フィールドは使わない → 実際のスポーン数を制御しない。durationの自動計算にのみ影響し、`groupSize`未指定だと1体ずつのグループになってしまう
- `duration`の省略 → 後半グループが出ない、または（wave内で）護衛到達まで無制限スポーンし続ける不安定な状態になる

**時間軸の計算式（★最重要・今日何度も訂正が入った箇所）**

1. **`spawnInterval`の起点**：「あるグループの全個体スポーン完了 → 次グループ開始」の間隔（グループ開始〜開始ではない）。`WaveManager.js:85-86`で確認済み

2. **`duration`と`startDelay`は直列**（同じ基準点からの並行カウントではない）：
   ```
   wave開始（前waveの終了 or ステージ開始）
     → startDelay 待機
     → elapsed = 0 スタート
     → duration 秒後、このwaveは終了
   ```
   ⚠️ **重要**: wave[1]以降のstartDelayは「そのwave内の最初のグループ出現を遅らせる」だけであり、
   waveの終了タイミング（elapsed基準のdurationカウント）には影響しない。
   startDelay が duration に近い値だと、wave内でほとんどゾンビが出ない可能性がある。

3. **wave内、numGroups（グループ数）を指定するフィールドは存在しない**。グループはelapsed=0基準で `spawnInterval` 間隔に発火し、**「発火判定時のelapsed < duration×1000」なら発火する**。つまりnumGroupsは`duration`と`spawnInterval`から決まる。設計者が「何グループ出したいか（N）」を決めたら、以下の**上下両側の不等式**で`duration`を挟む：
   ```
   groupSize=1 の場合（発火はelapsed = 0, si, 2si, ... で起きる）:
     N番目のグループ発火時刻 = (N-1)×spawnInterval
     N+1番目のグループ発火時刻 = N×spawnInterval

     (N-1)×spawnInterval < duration×1000 ≦ N×spawnInterval
   ```
   ⚠️ **【最重要・2026-07-02にoff-by-oneで事故】** 上限側（`≦ N×spawnInterval`）を必ず守ること。
   duration がこれを超えると **N+1番目のグループが「duration境界の直前」で駆け込み発火し、意図より1体多く湧く**。
   例：spawnInterval=5000, N=4体狙い → `15000 < duration×1000 ≦ 20000`、つまり **duration=16〜20秒**。
   誤ってduration=24秒にすると、elapsed=20000ms<24000msで5体目が湧いてしまう（実際にこれで+1体事故）。
   groupSize≧2の場合は各グループ内の最終個体時刻 `(groupSize-1)×groupInterval` も加味すること。

4. **wave全体の絶対終了時刻（wave indexが切り替わる時刻）**：
   ```
   ゲーム開始を t=0 として、

   wave[0] が終了する絶対時刻 = startDelay₀ + duration₀×1000
   wave[1] が終了する絶対時刻 = startDelay₀ + (duration₀ + duration₁)×1000
   wave[n] が終了する絶対時刻 = startDelay₀ + Σduration[0..n]×1000
   ```
   ※startDelay は wave[0] のものだけが累積に含まれる。wave[1]以降の startDelay は
   あくまで「そのwave内で最初のグループが出るまでの待機」であり、wave終了タイミングには加算されない。

   ⚠️ **【最重要・最終waveの罠】** `_calcWaveIdx()`のループが `waves.length - 1` までしか回らないため、
   **最終wave（配列の最後）はdurationで終了しない。護衛が到達（escort.reached）するまで無限に発火し続ける。**
   最終waveの`duration`値は実質無意味。よって最終waveの敵数は「発火間隔」と「護衛到達までの残り時間」から自動的に決まる。
   設計時は「最終waveは護衛到達まで敵が途切れず湧く」前提で組むこと（敵数を厳密に固定したいなら最終waveの後にダミーの空waveを置く等の工夫が要るが、通常は割り切ってよい）。
   最終waveの実発火数 = floor((護衛到達時刻 − 最終wave開始時刻) ÷ spawnInterval) + 1

5. **護衛到達前に全waveが湧き終わるかの検算は必須**：
   ```
   startDelay₀ + Σduration[全wave]×1000 < 護衛到達時間(ms)
   ```
   これを満たさないと、後半の敵（特に最終waveの終盤）が出現しないまま切り捨てられる

6. **現状の難易度設計方針（Plan A）**：wave内の敵は全員同じ強さ。escalation（強くなる）はwave間の敵種切り替えでのみ行う。wave内で敵を強くする設計（Plan B）は現状シミュレーターなしでは危険（C-3参照）

7. **【実装済 2026-07-02】`zombieSpawns`への敵種紐付け（`spawn.type`）**：
   ```json
   { "col": 5, "row": 9, "type": "police" }
   ```
   - `type`未指定 → 従来通り`wave.enemy`に従う（後方互換）
   - `type`指定あり → `ZOMBIE_BASE`テーブルからHP/speed/skinを丸ごと上書き
   - 新type: `salaryman`（normalの後継・基準値）/ `worker`（altの後継）/ `police`（新規）。既存の`normal`/`alt`系typeも後方互換で残存
   - **仮置き数値（要チューニング、G章参照）**：

     | type | HP | speed | skin |
     |---|---|---|---|
     | salaryman | 75 | 10 | salaryman |
     | worker | 100 | 9 | worker |
     | police | 150 | 12 | police |

     ⚠️ policeが最速最硬（上位互換気味）になっている。タンク役を作るなら意図的にクロスさせる調整が要る

### A-5. zombieSpawns

- 護衛が近づくたびに**「護衛から最も近いspawn地点」がFlowField距離で自動選択**される（ランダムでも順番でもない）
- 1グループの敵は全員同じ1箇所から連続して湧く
- **制約**：ゴール地点・path上のセルに置かない
- **スポーン後2秒の幽霊状態**（`Zombie.js` `_spawnTimer=2000`）：スポーン直後の2000ms間、ゾンビは完全無敵（移動不可・護衛への攻撃不可・タワーからの被弾不可、alpha0.4の半透明表示のみ）。2秒経過後に通常のゾンビとして動き出す。プレイヤーへの視認猶予として機能する良い仕様だが、難易度計算では「敵の実効的な脅威開始が湧きの2秒後」になる点に注意（G-5参照）

### A-6. buildSpots

- **必須**。空配列だとタワーを1基も設置できずゲームとして成立しない
- path・props(footprint)のセルと重複しないこと

### A-7. props（当たり判定あり・footprint注意）

- **propsは自動的に`obstacles`と同等の当たり判定を持つ**（`GameScene.js:54-61`）。`obstacles`配列への二重登録は不要
- タワー設置時も同様にブロックされる（`GameScene.js:946`）
- **footprint**：JSON上は左上1座標のみ指定。実際のブロック範囲は`PROP_DEFS`の`cols × rows`分、全セルが自動的に塞がれる

  | type | cols×rows |
  |---|---|
  | shop | 4×4 |
  | shop_front / shop_back / house_front / house_back / 家青 / 家緑 / アイスクリーム / ハンバーガー | 3×3 |
  | guardrail_h / guardrail_v / tree / 木1 / 木2 / ベンチ | 1×1 |
  | 公園 | 6×6 |
  | 駐車場 / 駐車場車有り | 4×2 |
  | スーパーマーケット | 6×3 |
  | 商店街 / house5_front / house5_back | 15×3 |

  ※新しいtypeが追加された場合は都度CCに`PROP_DEFS`の値を確認すること

- **⚠️既知の事故パターン**：**props配置に対するバリデーション（衝突警告）は一切ない**。path上やbuildSpots上にpropsのfootprintが被っても無警告でそのまま動く → 護衛が経路上でブロックされてフリーズ、buildSpotsが使用不可になる等のバグが発生する。**propsを置く際は必ずpath全座標・buildSpots全座標とのfootprint衝突を手動（または計算で）確認すること**

### A-8. ground_cells

- 道路などのテクスチャ指定のみ。**当たり判定には一切影響しない**（見た目だけ）

### A-9. minSurvivors / クリア・敗北判定

- 全護衛の移動が終了した時点で、`survivors（ゴール到達数） >= minSurvivors` ならクリア
- **即敗北判定あり**：道中で「残っている護衛が今後全員ゴールしても`minSurvivors`に届かない」と判明した瞬間、即座に敗北になる（ゲームが最後まで進行しない）

### A-10. 護衛variant拡張（son / grandma / cat）【実装済 2026-07-02】

- HP/speedは元々完全にJSON側で指定する仕様（コード側にvariantごとのデフォルト値なし）。この部分は今回無変更
- `GameScene.js`の`VARIANT_NAMES`（リレー表示名）に`son`・`cat`を追加登録
- `Escort.js`の`ESCORT_SPR`に`son`/`grandma`/`cat`のスプライト定義を追加（**frameH・scale・ox・oyは仮値**。アセット未確定のため、画像が揃い次第実測値に更新が必要）

- ⚠️ **son/grandma/catの実際のHP/speed数値はまだ未決定**。コード側の受け皿ができただけで、ゲームデザインとしての数値はこれから（G章で管理）

## C. 未実装の設計方針（ロードマップ・現在の実装ではない）

### C-1. ★スコア式（ドラフト、未実装）

- 複数護衛ステージ：`(生存者数÷全体人数)×10点 + (生存者の平均HP%)×10点`（0〜20点）
- 単独護衛ステージ（チュートリアル想定）：残HP%のみで判定
  - 70%以上 → ★3 / 30〜70% → ★2 / 30%未満(生存)〜ギリギリクリア → ★1
- ⚠️ **2026-07-02追記**：この閾値自体は変更なしだが、C-5のレベリングシステム導入により「★3をいつ取れる想定か」の前提がC-2から変わっている。式自体は流用可

### C-2. 難易度カーブ目標（ドラフト・2026-07-02 全面改訂）

**旧版（破棄）**：「1〜5ステージは初見で★3到達率80〜90%」という基準だったが、C-5のタワーレベリング導入に伴い前提が崩れたため破棄。

**新版の考え方**：
- タワーはクリア後の★でレベルアップし、恒久強化される（C-5参照）。そのため「★3到達率」は初見の難易度指標として機能しなくなった
- 管理指標を**★1到達率（＝クリア率）**に主軸を移す。★3到達率は「Lv1タワー基準」と「レベリング後」を分けて考える必要がある

| 指標 | 目標 | 備考 |
|---|---|---|
| ★1到達率（Lv1タワー、初見） | 全ステージ通して高め（例:80%以上） | 「スタートからクリアできない」を避けるための下限ライン |
| ★3到達率（Lv1タワー、初見） | **低くていい（意図的に）** | 序盤ステージ含め、Lv1では★3が基本届かない設計。効率配置を見抜けた人だけが到達できる腕試し |
| ★3到達率（レベリング後） | 事実上ほぼ100%に近づく想定 | 強化すれば取れるようになるのが前提なので、目標値として管理する意味は薄い |

- ⚠️ **序盤ステージ(1〜5)であっても★3はLv1では基本取れない設計にする**。これは「簡単すぎる」ことを意味しない。★1(クリア)のハードルは低いまま、★3(完璧)のハードルだけ高く保つ、という**閾値間の差を広げる**設計
- **3つの閾値（★1/★2/★3）を同時に満たす調整は、机上の計算だけでは決まらない**。1回の設計で正解に到達する前提を持たず、**複数回のプレイテストサイクルを前提に運用すること**（D-3チェックリストにも反映予定）

### C-3. wave内escalation（Plan B、保留中）

- 現状（Plan A）：wave内は均一、wave間でのみ強くなる。切り捨てが起きても「数が減る」だけで質的崩壊は起きない
- Plan B（wave内で敵を差し替えて強くする）へ移行する場合、「締めの強敵だけ丸ごと消える」事故が致命的になるため、**移行前に切り捨て検出シミュレーターの実装が必須条件**

### C-4. プレイテストのバイアス問題

- 現在のテストプレイは小松さん本人（TDジャンル熟練者）のみで行われている。効率配置を即座に見抜けるため、**このテスト結果は「上級者にとっての難易度」であり、初心者の難易度感とは乖離しうる**
- 対策候補：意図的な非効率配置でのテスト、外部プレイテスターの活用、Claudeによる「初心者ならどう置くか」の予測シミュレーション

### C-5. タワーレベリング・★貯金システム（ドラフト、未実装、2026-07-02決定）

- タワーは**個別（そのステージ内の1基）ではなく、タワー種全体**がクリア後の★でレベルアップする、メタ進行（ローグライク的な恒久強化）
- レベルアップの対象・タイミングはプレイヤー任意。ステージクリア後に★を消費してどのタワー種を上げるか選べる想定
- **レベルダウンも可能で、★は「貯金」として扱う**：ダウンすると使った★がストックに戻り、上げると再度消費される。上げ下げに心理的コストがかからない設計（恒久ロスがない）
- **この仕組みが持つ2つの機能**：
  1. やりこみ要素：レベルを落として縛りプレイ（例:Lv1でノーダメクリアを狙う）ができる。数値化・共有しやすい目標になる
  2. 下手ユーザー救済：初見でクリアできなくても、レベリングを重ねれば後から★3を取れるようになる
- **死にタワー対策としても機能する狙い**：複数護衛が同一経路を通る設計（A-3）と組み合わせ、短射程/長射程それぞれに出番を作る。ただし基礎値バランス（G章、短射程は威力強め・長射程は威力控えめにする案）との併用が前提
- ⚠️ **序盤ステージへの再訪価値の問題**：育ちきったプレイヤーが序盤ステージに戻ると瞬殺できてしまう。★3をLv1では基本取れない設計にすることで、「レベルを落とさないと★3が取れない」＝**常設チャレンジとして再訪価値を保つ**方針にした（C-2参照）

## D. 作業フロー

### D-1. 推奨フロー（今日確立した型）

1. **小松さんが道路込みで街を整形**（`ground_cells` + `props`）。道路レイアウトがそのまま護衛の経路の骨格になる
2. Claudeに渡す
3. Claudeが道路レイアウトから`path`座標を書き出す
4. Claudeが`zombieSpawns`・`buildSpots`をpath/props全footprintとの衝突チェック込みで計算
5. Claudeが護衛到達時間を逆算し、A-4の式に従ってwave設計（数値の逆算・検算）
6. 小松さんがプレイテストしてHP%・被弾数などをフィードバック
7. 必要に応じて数値だけ再調整（街の形は基本触らない）

※逆に「Claudeが先にpath/buildSpots/spawnを決めて、後から街を当てはめる」フローは、街のためのスペース計画が存在しないため窮屈になりやすい（stage_01で実証済みの失敗パターン）

### D-2. 役割分担

| 役割 | 担当 |
|---|---|
| エンジンの実際の挙動（コード事実） | CC |
| 数値検証（時間逆算・衝突チェック・構造的副作用の指摘） | Claude |

## E. 既知の落とし穴（今日発生した実例ログ）

1. **セルサイズの旧仕様混在**：メモリ上「CELL=64px」が残っていたが実際は100px。数値計算前に必ずconfig.jsの現在値を疑うこと
2. **`count`フィールドの罠**：wave定義に`count`があっても実スポーン数を制御しない。`groupSize`なしだと1体ずつの弱いグループになる、またはduration次第で無制限スポーンになる
3. **`spawnInterval`の意味誤読**：グループ開始〜開始の間隔だと誤解しやすいが、正しくは「前グループの湧き終わり→次グループ開始」
4. **`duration`と`startDelay`の関係誤読**：同じ基準点から並行カウントだと誤解しやすいが、正しくは直列（startDelay消化後にelapsed=0）
5. **grid10×8で街が窮屈になった事例**：チュートリアルステージでも街としての説得力を求めるなら15×10クラスが実質的なミニマムサイズだった（stage_01の反省）
6. **propsの無警告衝突リスク**：footprintの概念を知らずに配置すると、経路ブロック等の重大バグが無警告で発生する
7. **path区間数の数え間違い**：長いpath（40区間超）を目視でフェーズごとに数えると1区間分ズレやすい（今日、43区間と数えたが実際は44区間だった）。護衛到達時間がズレるとwave設計全体の検算が狂うため、区間数は`path.length - 1`で機械的に数えること（目視カウントに頼らない）
8. **durationのoff-by-oneで+1体**：「duration ≧ 最終スポーン時刻」だけ守っても、上限を超えると次グループが駆け込み発火する。設計値4体・spawnInterval5000・duration24秒 → elapsed=20000ms<24000msで5体目が湧いた実例あり。**必ず `(N-1)×si < duration×1000 ≦ N×si` の両側で挟む**（A-4-3参照）
9. **最終waveが護衛到達まで無限湧き**：`_calcWaveIdx()`が`length-1`までしか判定しないため、最終waveはdurationを無視して護衛到達まで湧き続ける。設計値12体のwave2が実際は14体湧いた実例あり。最終waveの敵数は「発火間隔と護衛到達までの残り時間」で決まると理解して設計すること（A-4-4参照）
10. **HP増だけでは難易度が上がりにくい**：敵speedが半減している環境ではタワー射程内の滞在時間が2倍になり、敵HPを1.5〜1.8倍にしてもタワーが処理しきってノーダメになる（v3で実証：helmet HP60→110でも4基ノーダメ）。1体ずつ湧く限り処理が溢れないため、難易度を上げるにはHPよりも`groupSize`（同時出現数）を増やして瞬間的な処理能力オーバーを作る方が効く

   - 距離 = 44 × 100px = 4400px
   - 到達時間 = 4400 ÷ 80 × 1000 = **55000ms**

2. **wave0の検算**（groupSize=1なので個体数=グループ数）
   - group-formula = (numGroups-1) × spawnInterval = 3 × 2500 = 7500ms
   - duration₀ ≥ 7500ms → **8秒**に設定（バッファ500ms）
   - wave0終了(絶対) = startDelay₀ + duration₀×1000 = 6000 + 8000 = **14000ms**

3. **wave1の検算**（n≥1のstartDelayはwave内の最初のグループを遅らせるだけ、終了時刻には加算しない点に注意）
   - group-formula = (7-1) × 2000 = 12000ms

### G-5. 難易度レバーの効き序列（2026-07-02 実測で判明）

speed半減環境（護衛40・敵27.5〜35）では、**敵の射程内滞在時間が長くなり、多くの難易度調整が「滞在時間の余裕」に吸収される**。3回の実測（v2速度/v3のHP増/v4のgroupSize増）で全てノーダメだったことから、以下の効き序列が判明：

| レバー | 効き | 理由 |
|---|---|---|
| タワー`fireRate`延長 | **強** | 滞在時間内の手数を直接削る。滞在時間の有利を打ち消す唯一の直接手段 |
| タワー`range`短縮 | **強** | 滞在時間そのものを削る |
| タワー購入価格↑ | **強（副作用小）** | 「いつでも強タワー」を防ぐ。強さを殺さず依存度だけ下げる |
| 敵`speed`↑ | 中 | 相対速度差を詰めるが「がちゃがちゃ感」再発リスク |
| startMoney↓ | 中 | ただし本作はタワー売買が基本運用（下記）なので、絞りすぎると機動的運用の楽しさを削ぐ |
| 敵HP / damage / groupSize | **弱** | 滞在時間に吸収されやすい。単体では効きにくい（複数レバー併用時のみ有効） |

- ⚠️ **スポーン後2秒の幽霊状態（A-5補足）**：ゾンビはスポーン後2000ms完全無敵（移動・攻撃・被弾すべて不可、半透明表示）。この2秒＋低速移動が「処理猶予」を生み、groupSize増が効かない一因。プレイヤーの視認猶予として良い仕様だが、難易度計算では敵の実効的な脅威開始が2秒後ろ倒しになる点に注意
- **タワー売買が基本運用**：本作はタワーを設置しっぱなしにせず、護衛の移動に合わせて売却→再設置する動的運用が前提（`sell`列＝売却額）。エスコート型TD特有の「守るべき場所が時間で動く」性質に対応。所持金の潤沢さ＝置き直しの自由度なので、難易度レバーとして所持金を使う際はこの楽しさとのトレードオフに注意

### G-6. タワー性能調整方針（2026-07-02 議論、未実装）

- 目指すのは「狙撃を弱くする」ではなく「**他タワーに狙撃にない固有役割を与える**」方向（KRのタワー役割分化を参考）。狙撃が複数レーンをカバーできる強さ自体はTDの核心的な面白さであり、否定しない
- KR流の王道は「**長射程＝低速・単体特化 / 短射程＝高速・数処理**」。狙撃無双を崩すなら、damage削減より`fireRate`延長で「手数を奪う」方がジャンル的に正しい
- ただしfireRateを延ばしすぎると「撃たない待ち時間」がストレスになるため、極端にはしない
- 具体案として挙がった狙撃調整（いずれか単独から試す。同時適用は過剰弱体化リスク）：
  1. range 4.5→3.0〜3.5に短縮（2.0まで一気に詰めると狙撃自体が死にタワー化するので段階的に）
  2. 購入価格 200→350〜400（最優先・副作用最小。「いつでも狙撃」を防ぐ）
  3. damage弱体化で3〜4発撃破に（単体では弱いが1と併用で効く）
- **推奨手順**：まず案2（価格）だけ適用してプレイ→まだ狙撃一強なら案1か3を一つ足す（変数を同時に動かさない）

### G-7. タワー2種限定フェーズ（2026-07-02 決定）

- **難易度計測フェーズの間は、タワーを「通常(basic)」「狙撃(sniper)」の2種のみに限定する**
- 理由：タワー7種あると「どのタワーを何基置いたか」でプレイ結果がブレて、ステージ側の難易度なのかタワー選択なのか切り分けられない。変数を減らして難易度を確定させるため
- basic（range2.5/fireRate900/dmg25＝手数・前線処理）と sniper（range4.5/fireRate1600/dmg調整中＝一撃・複数レーンカバー）の最小対比構成でステージ難易度を測る
- 残り5種（rapid/laser/punch/bat/cannon）は、この2種の基準が固まった後に「2種に対してどういう固有役割か」を持たせて1つずつ追加する
