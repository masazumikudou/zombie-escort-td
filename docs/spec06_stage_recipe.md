# ステージ設計レシピ v1.0（2026-07-18）

**この文書の役割**: 新文法（区間制・initial待機＋盤外侵入・護衛範囲円）でステージを設計するための完全な手順書。stage_honban_test.json（型見本）の設計過程で確定した手順と、踏んだ地雷の実例をすべて記録する。P3量産はこのレシピに従えば、軽量モデル＋CC体制で回る。

**正典の位置づけ**: 工程表・spec05人格章・区間スキーマ・構成レビューに続く5点目。設計判断で迷ったら人格章（なぜ作るか）→本レシピ（どう作るか）の順で参照。

技術的な低レベルルール（prop衝突・PROP_DEFS・タワーコスト・シムCLI構文・旧文法の確定ステージ）は [spec05_stage_design.md](spec05_stage_design.md) を参照。本レシピはそれを前提にした新文法（区間制）専用の設計工程ガイド。

## 0. 設計の大原則（人格章からの継承）

- このゲームは「家族を読みで守り抜く、静かな脱出劇」。ステージの面白さの核は**読み**
- 見える脅威(initial)=初見の読み／見えない脅威(侵入)=周回の読み。三層(良/並/雑)は上達の時間軸
- 良=初見ほぼ不可能で正しい。並プレイ・強化ゼロでCLEAR成立がバランスの下限（Q2原則）
- 成立(三層)と面白さ(実機の体感)は別物。シムは壊れてないことの担保、面白さの最終判定は常に実機

## 1. 設計の順序（この順を崩さない）

1. **マップ・導線・交戦地**（小松担当）——道路網、護衛のpath、山場にしたい場所
2. **護衛の時刻表を作る**（§2）——全設計の座標軸。これを最初に作らないと以降が全部狂う
3. **区間を切る**——山場のまとまりごとに2〜4区間。区間ごとに面白さの仮説を1個持たせる（§6）
4. **脅威を配分する**（§3）——見える(initial)と見えない(侵入)の使い分け
5. **侵入の発火を逆算する**（§4）——時間計算。ここを怠ると敵は全員すり抜ける（実証済み）
6. **buildSpotsを引く**（§5）——道路沿い原則ぜんぶ。機械チェック必須
7. **タワー3人格パターンを組む**（§7）——良/並/雑
8. **シム三層＋CLOSE_CALL**——数字の確認
9. **実機**——面白さの判定。区間ごとに「何を感じたか」を聞く

## 2. 護衛の時刻表（全設計の座標軸）

護衛は決定論的に動く。pathの累積距離÷speedで、各waypointの通過時刻が確定する。

```
時刻(秒) = 累積距離(px) ÷ 護衛speed(px/s)
```

**stage_honban_testの実例**（dad speed50）: A(14,3)=28s / B(14,9)=40s / C(3,9)=62s / D(3,15)=74s / ゴール=106s

この表が「動かせない時刻表」。敵・タワー・演出のすべてはこの時刻表に合わせて設計する。計算はPythonスクリプト（cum距離の積算）で機械化済み——R-6の検証スクリプトに統合予定。

## 3. 脅威の使い分けの法則

| | initial（見える脅威） | 盤外侵入（見えない脅威） |
|---|---|---|
| 時間計算 | **不要**（最初からそこにいる） | **必須**（§4） |
| 読み | 初見から読める（観察） | 周回で読める（記憶） |
| 配置の自由度 | 高い。導線のどこでも | 低い。交戦地から近い盤外限定 |
| 難易度レバー | 導線への近さ×数 | 数×発火密度 |

**法則1: 交戦地から遠い脅威はinitialで置け。** 盤外から鈍足(30px/s)で長距離を歩かせる設計は時間的に破綻する（§9地雷3）。盤外侵入のスポーン地点は目安として**交戦地から5セル以内の盤外**。

**法則2: 追わせるな、向かわせろ。** leash待ち伏せは精密な時間逆算が必須で壊れやすい。**護衛の進行方向の前方から来る敵（直接追跡・正面衝突）は閉速度=護衛speed+敵speed で必ず出会う**ため、時間設計がほぼ不要。侵入の基本形は正面衝突、leashは交戦地至近の盤外からの短距離に限る。

**法則3: 背後からの追走は永遠に追いつかない。** 敵speed(30) < 護衛speed(50)。護衛が通過した後に後方から出す敵は、直接追跡では絶対に接触しない。出すならleashで先回りさせるしかなく、それには法則1の距離制限がかかる。

## 4. 侵入の発火逆算（必須計算）

```
敵到達時刻 = 発火時刻 + 盤外歩行時間 + 盤内歩行時間
           = 護衛が発火wpを通過する時刻 + (スポーン地点から交戦地までの距離 ÷ 敵speed)
成立条件: 敵到達時刻 ≦ 護衛到達時刻 - マージン(2〜4秒推奨)
```

- マージンは「敵が先に着いて待ち構える」余裕。0だとすれ違いギリギリで不安定
- **盤外部分の歩行時間を忘れるな**（盤外2セル=200px=6.7秒。§9地雷5）
- 正面衝突型（法則2）はこの計算が不要。会合点だけ概算しておく: 会合時間 = 初期距離 ÷ (護衛speed+敵speed)
- 発火wpは必ず**自区間のrange内**に収める。範囲外のatWpIdxは発火しない（現実装。区間またぎ予約発火は未実装——必要になったらCCに依頼、ただし正面衝突型で大半は回避可能）

## 5. buildSpots設計

- **方針: 道路沿いの空き地は原則ぜんぶspot。** 選択肢の多さ=迷い=読みゲーの本体。少ないspotは答えを配っているのと同じで、「下手な配置」が存在できない
- 除外: propの**フットプリント**（登録座標だけでなく3×3等の足元全域。§9地雷6）、道路セル、グリッド外
- 間引き: 過度な隣接連鎖はタッチ誤爆リスク（正式ルールは2-5の実機タップ検証後）
- **山場を作る前に、そこに届くspotがあるか確認**（punchの射程100px問題、seg4右半分ノーカバー問題の教訓)
- 難易度レバー: Lv1はspot少なめ（迷わせない=チュートリアル）、Lv上昇で増やす
- 検証は機械チェック必須（R-6スクリプト）。目視は26%の衝突を見逃した実績がある

## 6. 区間設計と面白さの仮説

- 1区間=1つの面白さ仮説。stage_honban_testの実例: seg1=読み→増援 / seg2=緩の対照群 / seg3=不気味な静寂 / seg4=挟撃クライマックス / mom区間=キャスト入れ替え
- 仮説を区間に分離しておくと、実機の感想が「どの仮説が当たったか」の実験結果になる（1ステージ=5実験）
- 緩急: 全区間を賑やかにすると緩急が死ぬ。意図的な「緩」区間を入れ、実機で「心地よい息継ぎ」か「退屈」かを判定
- 区間境界のretreat（残敵の画面外退場）が失敗の蓄積を切る。区間内で解決させたい敵はその区間に置く

## 7. タワー3人格パターン（三層検証の組み方）

三層は難易度の層ではなく**プレイヤー像の層**。全パターン900G基準（予算は同額、使い方だけ変える）：

| 層 | 人格 | 組み方 | 期待される破綻 |
|---|---|---|---|
| 良 | 読み切る人 | 全区間に配分。各交戦地に届く最良spotを吟味 | 破綻しない（CLOSE_CALLは出る=ギリギリ） |
| 並 | 反応する人 | **前半の交戦地に過剰投入（同じ場所に2〜3基）、後半スカスカ** | 終盤の資金枯渇でじわじわ削られる |
| 雑 | 釣られる人 | 900G使い切るが、ice散財・交戦地から遠い「イメージ買い」・自陣固め | 序盤〜中盤で崩壊（ただし§9地雷7参照） |

**判定基準**: 良＝HP100%クリア可能な配置の存在が全ステージ共通の絶対条件（100%に必要なタワー総額もタイトさの物差しとして記録）。雑＝100%に届かないことが条件で、落ち幅がLv別目標帯に入るか確認（Lv1-2=HP60-80%生還/Lv3-5=HP10-40%/Lv6-8=GAMEOVER境界/Lv9-10=確定GAMEOVER）。雑=GAMEOVER必須は誤り（入門ステージが罰ゲーム化する）。並は合否判定から除外し観測のみ（終盤型破綻の診断データ）。難易度の数値定義＝「良100%−雑スコア」の差分がLv。CLOSE_CALL=体感の指標、雑落差=構造の指標の2軸で難易度を可視化する。

**CLOSE_CALLの目安**（仮当て・実機照合で更新すること）: 0回=楽勝すぎNG / 3〜10回=ノーダメギリギリ目標帯 / 20回超=修羅場すぎ。

## 8. 実機照合の聞き方

実機後の質問は「面白かった?」ではなく**「各区間で何を感じたか、一言ずつ」**。区間=仮説なので、答えがそのまま実験結果になる。シムのCLOSE_CALL分布と突き合わせ、「数字と体感の対応表」（§10）に1行追加する。

## 9. 踏んだ地雷集（すべて実話・2026-07-17〜18）

1. **initialの書式**: `{col,row,enemy}`直書きは**黙殺される**（エラーなし）。正: spawns辞書に名前付き地点を登録し`{spawn:"名前",enemy:{...}}`で参照。triggersと同形式
2. **leashToの位置**: 必ずenemyオブジェクトの**中**。外（トリガー直下）に置くと黙殺
3. **時間計算を忘れると全員すり抜ける**: 初版は5トリガー中5個が「敵到達>護衛到達」で、9体全員が交戦ゼロのすり抜け。§4の計算を必ず行う
4. **区間範囲外のatWpIdxは発火しない**: 前倒しした発火wpが自区間のfromWpより手前に出たら、正面衝突型への転換を検討（区間またぎ発火は未実装）
5. **盤外歩行時間の計上漏れ**: 盤外2セル分=6.7秒。地味に効く
6. **propはフットプリントで避ける**: 登録座標1セルだけ見て除外した結果、42spot中11箇所(26%)がめり込んだ。機械チェック必須
7. **buildSpotが多い盤面では雑配置も偶然当たる**: 雑パターンが偶然initial の真横に建って粘った実例あり。雑を組むときは「交戦地の近くに建てない」を意識的に。逆にこれは盤面の健全さ（どこに建てても多少は機能する）の証明でもある
8. **「シムが通る」と「実機の絵が正しい」は別**: カメラ拡張で虚無が見えた件、墓デカール自動追従の件。絵の要件はコードの理屈で先回りできない。毎回実機で見る
9. **同期漏れは全てを無に帰す**: npm run syncを忘れると実機は旧コードのまま動き続け、「実装したのに変わらない」の調査で半日消える。実機確認前のsyncをルーチン化
10. **予約発火は不可能・トリガーは交戦させたい区間に置く**: 「区間の最後のwpで発火→次区間で交戦」は構造的に成立しない。区間離脱時のretreat/pendingクリアに猶予がなく、発火直後（次wp到達=最短2秒）に敵が強制退場する。seg3の敵を3回連続で消し飛ばした実話。トリガーは必ず交戦を起こしたい区間そのものの範囲内(fromWp〜toWp)に置く
11. **leash待ち伏せは正面衝突を作れない**: leashで交戦地に足止めした敵は、待ち伏せ地点で護衛とすれ違った後、起動して護衛を追う=背後からの追走=永遠に追いつかない(法則3)。正面衝突を狙うなら**leashを外した直接追跡**で、敵を護衛の進行方向の逆側から出す(下降路なら下から、直進路なら前方から)
12. **タワー配置の入力書式とログ表示形式の混同**: run_sim.jsの入力書式は `type:col,row[@ms]`（コロン+カンマ、括弧なし。正規表現`^(\w+):(\d+),(\d+)(?:@(\d+))?$`）。一方[BUILD]ログの出力表示は`type@(col,row)`形式（`@`+括弧）で人間向けに整形されている。この2つを混同して出力形式のまま再入力すると、パーサーに一致せずタワーが1基も配置されないまま（[BUILD]行皆無）シムが実行され、原因不明のGAMEOVERを引き起こす。入力欄には常に`type:col,row[@ms]`形式を使うこと。

**この表の使い方**: 実機照合のたびに「体感」列を埋める。行が5〜10本溜まった時点で、「CLOSE_CALL何回・どの分布だと面白いか」の翻訳が確定し、以後はシムの数字だけで面白さの一次判定ができるようになる=量産の完成形。
