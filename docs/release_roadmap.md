# Zombie Escort TD — リリース工程表 v2.0（2026-07-17 / 新文法転換版）

**この文書の役割**: リリースまでの共有レール。僕（小松）・Claude・CC・Opus/Sonnetの誰が見ても「今どこ／残りいくつ」が分かるスタンプラリー。各フェーズに完了定義（＝スタンプ条件）を付す。期限は設けない（進捗はスタンプで測る）。

**関連ドキュメント（必読）**:
- **spec05_game_identity.md（ゲームの人格章）** — このゲームが何であるかの北極星。新機能・演出・調整の是非判断は必ずこの章との整合で行う。一枚看板は「家族を読みで守り抜く、静かな脱出劇」

- spec05（設計ガイド本体）

**v2.0の変更点（新文法転換・2026-07-17）**: スポーン文法を時間駆動（spawnEvents）から**区間制・位置トリガー（segments）**へ転換する実証実験を開始。1-4/1-4bを✅から差し戻し（旧Y実装のsetSpawnMultiplier方式は廃案）、1-4.5「新文法実証実験」を新設（撤退基準付き）。1-7を新文法版に差し替え。スタンプ進捗は11/55→9/55に後退（量産前に仕様の手戻りを検出できた健全な後退）。CLOSE_CALL指標実装済み。工程表の更新権限ルールを新設（本文末尾）。ゲームの人格を確定しspec05_game_identity.mdとして文書化。

**v1.9以前の変更履歴**: v1.9=倍速バグ修正（1-6完了）／v1.8=護衛個体パラメータ・HPバー修正（1-2完了）／v1.7=escortDef別wave・run_sim.js整備（1-1/1-8完了）／v1.6=設置コスト改定・アップグレードシステム実装（1-3完了）／v1.5=タワー実装完了確認・rangeTest再計算／v1.4=sniper射程修正450→350px／v1.3=Y実装完了（※v2.0で差し戻し）／v1.2=タワー5種体系確定／v1.1=CC実装レビュー反映

**ゴール**: Android買い切り300円・一発リリース。20ステージ＋家族リレー＋Y＋ボーナスステージ、フル搭載完成版。

**確定事項（ぶれない前提）**:
- 20ステージ・10段階難易度（stage_01=Lv5基準）
- 家族リレー＝基礎文法（MAX5人: dad/mom/son/grandma/cat、ステージごとに人数構成可変）
- クリア条件: 1人でもゴールでCLEAR／全滅NOT CLEAR。生存人数でポイント（＝メタ通貨）
- 後続の家族ほど重い（護衛パラメータ＋ゾンビ強さの2軸）
- **【v2.0転換中】スポーン文法: 区間制・位置トリガー（segments）へ転換の実証実験中（1-4.5）。GO判定で正式採用、NO-GOでspawnEvents（時間駆動）に復帰**
- **区間（シーン）制: リレー成立時に前区間の残敵は画面外へ退場（retreat）。失敗の蓄積は区間内に閉じる**
- Y（寄り道・静的TDモード）: ステージ内で家族1人だけ発動、誰が/有無は初見不明。**【v2.0で再定義】旧setSpawnMultiplier方式は廃案。Y地点紐付けの専用流入（yInflow）として1-4.5 GO判定後に再実装**。世界観上の意味づけ=家族が日常のつもりで寄り道（公園休憩・買い物）
- ステージ内タワーアップグレード採用（リレー成立の部品）
- タワーは新5種: normal/sniper/cannon/ice/punch。仕様確定・実装完了済み（【タワー確定パラメータ】参照）
- **採点思想（v2.0確定）: 生存人数が主軸。撃破数主軸は不採用（退場演出との矛盾・ノーダメ美学との不整合のため）。残マネーは初期リリースでは参考表示のみ。倒す快感は採点でなくゲームフィール（P5-4）で担保**
- メタ強化は「ご褒美」幅に絞る方向（Q2壁打ちで確定）
- 敵は4種: salaryman（実装済）＋鳥系（prop無視飛行）＋デブボス＋キックボード
- ボーナスステージ1種（ロジック流用ミニゲーム）——切り離し可能・最後尾
- 鉄則: シム=実機の完全一致（1×基準）。対症療法禁止。ズレは根本原因まで追う。**シム=シム一致（CLI/ブラウザ/単独実行/連続実行の一致）も同鉄則に含む**

### CLOSE_CALL指標（実装済み・v2.0新設）

「成立する」と「面白い」は別物。三層検証は成立の担保、CLOSE_CALLはギリギリ感の近似指標。

- 定義: ゾンビが護衛の至近（CELL×1.5=150px以内）まで接近してから撃破された回数。RESULT行に回数と最接近距離を集計
- 実装: Zombie.js（_closestToEscort）・GameScene.js・run_sim.js・simulator.html
- 目安（**※実機照合前の仮当て。後日の実機プレイで感覚と照合して基準に昇格させる**): 0回=楽勝すぎ・設計NG／3〜10回=ノーダメギリギリ・目標／20回超=修羅場すぎ・調整荒い
- 面白さの最終判定は常に小松の実機プレイ。CLOSE_CALLはその感覚を近似する道具にすぎない（spec05_game_identity.md参照）

### アップグレード仕様（v1.6実装済み）

3段階（Lv1=基準無強化・Lv2・Lv3）。Lv1→Lv2の費用 = 設置cost×0.5（Lv2→Lv3は未設定: null）。

|---|---|---|---|
| normal | ×1.15 | ÷1.15 | dmg・連射ともに向上 |
| sniper | — | ÷1.10 | 連射のみ（dmg・射程固定） |
| cannon | ×1.15 | — | dmgのみ（連射固定） |
| ice | — | ÷1.15 | 連射のみ（dmg固定） |
| punch | — | ÷1.15 | 連射のみ（dmg固定） |

> ※ fireRateMult は **間隔を割る**（÷）= 連射速度が上がる。シム: `upgrade:col,row@ms` で指定時刻にアップグレード実行。

### 設計原則（タワー設計・ステージ設計共通）

1. **単騎完結の禁止**: 1種のタワーだけでどんな交戦地も処理できる性能にしない
2. **交戦地の密度**: スポーン数1体を極力避ける（最低2体以上）
3. **アイスのシム扱い**: スロウ効果はシムに実装しない。三層検証はice抜きで成立させる。実機ではスロウが救済として機能
4. **AoEなし**: 範囲攻撃は実装しない
5. **パンチのbuildSpot**: 射程100pxのため、ステージ設計時に交戦地隣接の近接用buildSpotを意識的に確保すること（**stage_01・segment_test両方でpunchが届くbuildSpotが存在しないことが実測で判明済み。P3量産時の設計チェック項目に含める**）

### 工程表の更新権限ルール（v2.0新設）

工程表はリアルタイム更新とし、更新権限を以下の通り分ける:

**CCが直接更新してよいもの（機械的更新）**:
- タスクのチェックボックス（未完了→✅）
- スタンプ進捗カウンタの数字
- 「現在地」セクションの一行（「次は◯◯」）
- バージョン番号・日付・変更点欄への実装事実の記載

**CCが単独で書き込まないもの（判定・解釈を含む更新）**:

- スタンプ進捗が後退した場合の解釈・理由づけ

## リスク登記簿（工程を脅かす順）

1. **タッチ精度（CELL=100px×Android）** — 論理2000pxを物理画面に圧縮したときbuildSpotが指で押せない可能性。P3の20本を作ってからP6で発覚すると全やり直し。対策: 2-5でP3前にstage_01を実機/AVDで指タップ確認
2. **新文法転換の検証未了（v2.0新設）** — 1-4.5のGO/NO-GO判定が未確定のままP1後半に進むと手戻り。実機ビルドの新文法動作確認が直近のブロッカー。対策: 1-4.5で早期に撃ち合わせ、NO-GOなら即座に現行文法へ復帰（SegmentManagerは並存設計のため被害ゼロ）
3. **P1の仕様壁打ち不足** — Y再定義（1-4）の仕様が甘いままP3に入ると20本作り直し。対策: 1-7の全部載せ検証を必ず通す
4. **鳥系ゾンビ（1-5）** — 飛行セル次第でbuildSpot制約が変わりステージ設計に影響。対策: P3前に1体実装して挙動確認。重すぎれば「後半Lvのみ登場」で依存を切る
5. **P3の調整量** — シム工場があっても19本は量。対策: カーブ表(2-2)で仕様を先に固定し、1本あたりの意思決定を減らす
6. **メタ強化×難易度の掛け算（Q2）** — 対策: 「強化ゼロで全ステージ成立」原則を壁打ちで確定し、調整変数から外す
7. **シム2系統の乖離（v2.0新設）** — run_sim.js（CLI）とsimulator.html（ブラウザ）でスタブ・状態管理の片側修正事故が既に1回発生（Phaserスタブ漏れ・状態汚染）。対策: 修正は必ず両系統同時、可能ならスタブ・コアの共通化を検討（1-4.5クローズ後）
