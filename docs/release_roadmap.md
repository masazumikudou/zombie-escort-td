### 【P3前の議論事項】導線A/B振り分け設計

家族リレーで全員同一導線は体験が単調になるため、ステージ設計時に導線を複数本（A/B）引き、各家族に割り当てる方式を導入する（例: dad=A / mom=B / grandma=A）。S（スタート）とG（ゴール）は全家族共通、その間の経路のみがA/Bに分岐する。

**難易度レバーとしての意味**: 遺産タワー（前の家族が設置）は同一導線の次の家族には効くが、別導線の家族には射程外で無効化される。導線割り当て自体が難易度の設計図となり、資金継続仕様（escort交代でmoney継続）と噛み合う。同導線＝遺産が活きて楽／別導線＝遺産が死んで再構築を強いる。

**2つの導線パターン**:
- セパレート式: A/Bが最後まで交わらない。遺産タワーの有効/無効が完全に二値。家族ごとに独立した難易度を作りたいステージ向き。
- 交わり式: A/Bが途中で合流区間を持つ。合流点は全家族が必ず通る「一等地」となり、リレー全体で最も費用対効果が高い交戦地帯として機能する。資源配分の優先順位を問うステージ向き。合流点タワーもescortのspeed/HP差で「dadには足りるがmomには足りない」が起きうるため、一等地だが万能ではない詰めどころが生まれる（§4.5と整合）。

**データ構造**: エディター内部の編集モデルのみ変更（pathA/pathB共有配列＋各escortはlaneId('A'|'B')で参照）。**書き出されるステージJSON自体は変更なし**——各escortの`path`は従来通り独立した具体配列のまま（エディターがエクスポート時にA/Bの内容を該当escortへコピーするだけ）。ランタイム側（Zombie.js/GameScene.js/run_sim.js/simulator.html）は無改修。

**editor改修 — 実装済み（2026-07-24・CC実装、小松実機確認済み）**: 複数path描画（導線A=シアン・導線B=マゼンタで別色描画）・escort→導線割り当てUI（WAVE設定欄の「このWAVEの導線」ドロップダウン）・A/B両方ともSクリックから開始必須（既存の単一導線と同じ制約を踏襲）を実装。交わり式（A/Bの交差・重複）は専用の「合流点」定義UIを設けず、両レーンが独立配列のため同じマスを両方が通れば自然に交差する形で対応（明示的な合流点マーカーはなし）。既存ファイルの読込時は各escortのpath配列を比較し自動でA/Bを判別・復元する。

**位置づけ**: editorツール自体は完成。どのステージでセパレート式/交わり式のどちらを使うか、遺産タワーの設計への活かし方は引き続き小松判断（次の本番ステージ3-xから適用予定）。ステージ量産（P3）着手前に正式議論する。P2（2-2カーブ表の導線構成割り付け）と直結。

### 【P3方針・v2.5確定】逆順生産・ベースMAP方式（小松判断）

**逆順生産**: ステージ20（Lv10・難易度MAX）を最初に作り、そこから引き算で下位ステージへ降ろす。天井を先に実在させることで、以降の設計判断は「何を抜くか」に単純化される（積み上げ式だと天井の姿が最後まで見えず、後半で辻褄合わせが発生しやすい）。1-7（統合検証ステージ）がステージ20のプロトタイプを兼ねる。

**ベースMAP方式**: ベースとなるMAPを数種類作り、そこから複数ステージを派生させる。
- **固定（ベースMAPで決まる）**: 道路網・導線の骨格、プロップの配置枠（位置・footprint）
- **可変（ステージごとに変える）**: 敵の構成、buildSpotの配置、導線A/B割り当て、プロップの絵柄

**プロップ差し替え原則**: プロップは同サイズ（同footprint）での絵柄差し替えを前提とする。footprintが変わらないため、buildSpotやゾンビの通行判定への影響なしに見た目だけを変えられる。同じベースMAPでも見た目の異なる複数ステージを低コストで作れる。

**要検討（P3着手前）**: ベースMAP数種の具体案、buildSpot可変域の設計ルール（ベースMAPのどこまでを可変にしてよいか）。
