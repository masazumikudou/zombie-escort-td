## 1. 絶対ルール（破ると実機で壊れる）

### 1-1. prop フットプリント上にタワー・buildSpot を置かない

prop の footprint = ステージ JSON の `col,row` を左上として `cols × rows` 分のセル全部。

```
例: ハンバーガー(5,10) / cols:3, rows:3
    → col5-7, row10-12 が全部ブロック
```

**木2（1×1）も prop なので衝突する。** 視覚的に小さいが footprint は 1 セル占有している。

### 1-2. ゾンビは道路セル（ground_cells）上のみ移動（road-only）

GameScene.js と run_sim.js の両方で実装済み。`ground_cells` が存在するステージでは、リスト外のセルはすべて `pf.blocked` に追加される。

- ゾンビの移動は `ground_cells` に列挙されたセルのみに制限される
- スポーン座標・leashTo・護衛 path の全セルが `ground_cells` に含まれること（含まれない場合はFlowFieldが機能しない）
- 護衛 path の col14(row4-9)・row9横断(col4-7,col9-12)・row3(col4) も登録済み
- row16 (col3/col8/col13) を追加し下段(row17-19)と主マップを接続済み

### 1-3. シムが衝突の最終ゲートキーパー

`run_sim.js` はタワー配置時に `pf.isWalkable()` でチェックし、prop フットプリント内なら **`process.exit(1)` で強制停止**する。  
シムを通れば実機でも問題ない。シムを通らないパターンは実機に持ち込まない。

```
[ERROR] prop衝突: sniper@(5,10) はpropフットプリント内です。配置不可。
```

## 3. 経済設計（ステージ難易度の根拠）

### コインの流れ
- **収入**：初期資金（`startMoney`）＋撃破報酬（enemy.reward）
- **支出**：タワー建設（即時 or `@ms` 遅延）
- 時間ベースの自動収入なし（足止め金策が最適解化するのを防ぐため）

### タワーコスト（現行 balance.json 準拠）

| type | コスト | 射程（セル） | 備考 |
|---|---|---|---|
| basic | 100 | 2.5 | 汎用 |
| rapid | 150 | 2.0 | 速射 |
| sniper | 200 | 4.5 | 長射程・遅射 |
| laser | 350 | 999 | 全射程 |
| punch | 120 | 1 | 近接 |
| bat | 180 | 1 | 近接 |

> **数値の正典は `balance.json`。** ステージ検討時はここの数値を参照。コードに数値を書かない。

### startMoney 設計の目安
- 開幕即置き 3 タワーが限界になる程度が中級難易度
- 例：sniper(200)×3 = 600。startMoney:900 なら残り300のバッファ

## 6. ステージ JSON テンプレート構造

実物は `stages/stage_01_meander_verified.json` を参照。構造の骨格は以下の通り。

```json
{
  "id": "stage_XX",
  "name": "ステージ名",
  "grid": { "cols": 20, "rows": 20 },
  "startMoney": 900,
  "minSurvivors": 1,
  "escorts": [{
    "variant": "dad",
    "startTrigger": "afterPrevious",
    "start": { "col": 0, "row": 3 },
    "goal":  { "col": 19, "row": 15 },
    "path": [ { "col": 0, "row": 3 }, ... ],
    "hp": 100,
    "speed": 40
  }],
  "obstacles": [],
  "spawns": {
    "A": { "col": 19, "row": 3 }
  },
  "spawnEvents": [{
    "time": 15.3,
    "spawn": "A",
    "enemy": { "type": "salaryman", "hp": 200, "speed": 30, "damage": 10, "reward": 20 },
    "count": 2,
    "interval": 1000,
    "leashTo": { "col": 14, "row": 3 }
  }],
  "buildSpots": [
    { "col": 12, "row": 4 }
  ],
  "encounterHints": [
    { "col": 14, "row": 3, "note": "交戦地A（角）" }
  ],
  "ground_cells": [ ... ],
  "props": [
    { "type": "家青", "col": 0, "row": 5 }
  ]
}
```

**spawnEvents キー:**
- `time` : 秒単位（例: 15.3 = 15300ms）
- `leashTo` : エスコートの経路上の追跡開始地点。省略すると演出スポーン扱い
- `count` : 出現数。複数のとき `interval`（ms）で間隔指定

## 9. 演出スポーン設計ルール（leashTo なし・FlowField 追従型）

F1/F2/F3 のような「leashTo なし」の演出スポーンは FlowField で護衛を追う。  
護衛は **時刻ベース計算**（`_walkOriginTime` 方式）で移動するが、FlowField 追従型ゾンビは補正対象外なので、設計側で余裕を持たせること。

### 基本ルール

> **演出スポーンが護衛に「到達できてしまう」タイミングを、護衛 REACH 理論時刻より前に作らないこと。**

- leashTo なしゾンビは FlowField → 積分移動。フレームレートや速度倍率による位置誤差が残る
- `leashTo あり`ゾンビは時刻ベース計算のため誤差ゼロ。演出スポーンのみに本ルールが適用される

### 設計検証の式

```
演出ゾンビが護衛に接触可能な最早時刻 > 護衛の REACH 理論時刻
```

簡易計算（護衛が直進経路を等速でゴールする前提）：

```
護衛 REACH 理論時刻 = 経路総距離px ÷ 護衛速度 × 1000  (ms)
演出ゾンビ接触最早時刻 = スポーン時刻 + スポーン地点～護衛経路最近点の距離 ÷ ゾンビ速度 × 1000
```

### 推奨バッファ

| 状況 | 余裕の目安 |
|---|---|
| 護衛ゴール付近にスポーン | 護衛 REACH 後に接触する距離に設置（最低 1 秒分 = speed×1000ms） |
| 演出でゾンビが「惜しくも届かない」を演出したい場合 | シムで HP=100% CLEAR を確認してから採用 |
| 護衛のフレームレート依存誤差の余裕 | 護衛経路のセグメント数 × 16ms（stage_01 では +848ms 相当） |

### シム確認手順

1. `run_sim.js` を実行して演出ゾンビが護衛に攻撃していないことを確認
2. `[ESCORT_AT]` ログと `[ATTACK]` ログのタイムスタンプを比較
3. 実機で 1× 通常速度でも同じ結果になることを目視確認

### stage_01 における F1/F2/F3 の設計根拠

| ゾンビ | スポーン時刻 | 役割 | シム結果 |
|---|---|---|---|
| F1 | 105s | 演出（南西から接近） | 護衛 REACH 後に接近 → 攻撃不可 |
| F2 | 108s | 演出（南中から接近） | 護衛 REACH 後に接近 → 攻撃不可 |
| F3 | 111s | 演出（南東から接近） | 護衛 REACH ぎりぎり前（フレームレート依存） |

> **F3 は設計上グレーゾーン。** 実機 1× で HP=80% になることを確認済み（F3 が届いた）。  
> 今後新ステージで同様の演出をする場合は、REACH から最低 1 秒以上の余裕を設けること。
